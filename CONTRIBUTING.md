# Contributing to RLCatan

Thank you for your interest in contributing to RLCatan.

This project was developed as a software engineering capstone focused on reinforcement learning for Catan. Contributions are welcome, especially bug fixes, documentation improvements, test improvements, performance work, and UI or developer experience enhancements.

## Before You Start

Please:

- Read the project `README.md` for setup and usage instructions
- Check existing Issues and Pull Requests before starting work
- Open an Issue first for large changes, architectural changes, or feature proposals
- Keep contributions focused and reasonably scoped


## Coding Guidelines

Please follow these guidelines when contributing:

- Keep changes small and focused
- Prefer clear, readable code over clever code
- Match the style and structure of surrounding code
- Add comments only where they improve clarity
- Avoid unrelated refactors in the same pull request
- Update documentation when behavior, setup, or usage changes
### Python
- Follow PEP 8
- Write modular code with clear function and variable names
- Add or update tests for behavior changes where practical
### Frontend
- Keep components focused and reusable
- Follow the existing TypeScript/React project structure
- Test UI changes when practical and keep styling changes contained

## Pull Request Guidelines

When opening a pull request:

- Use a clear and descriptive title
- Explain what changed and why
- Link the related Issue if there is one
- Include screenshots or short recordings for UI changes
- Mention any setup, migration, or testing notes reviewers should know
- Keep the PR focused on one logical change

A good pull request description usually includes:

- Summary: what this PR changes
- Reason: why the change is needed
- Testing: what you ran to verify it
- Notes: any follow-up work, limitations, or reviewer context

### Commit Messages

Please use clear commit messages that describe the purpose of the change.

Examples:

- `Fix PPO action filtering for invalid trades`
- `Add tests for explanation API route`
- `Update installation instructions for Docker compose`

## Reporting Bugs
When reporting a bug, please include as much of the following as possible:

- What you expected to happen
- What actually happened
- Steps to reproduce
- Screenshots, logs, or error messages
- Operating system and environment details
- Whether the issue is in backend, frontend, training, or Docker setup

## Suggesting Features

Feature suggestions are welcome, especially if they are:

- Clearly motivated
- Well scoped
- Aligned with the goals of the project
- Feasible within the current architecture

For larger features, please open an Issue before starting implementation.
